# Executive Summary

Agentic coding systems must manage ever-growing histories within fixed context windows. Frameworks like OpenCode, LangChain’s DeepAgents, and similar multi-agent platforms use **context compression** to avoid token bloat.  In practice, tool outputs (like file contents or command results) **do count as context tokens** if fed back to the model – unless the system offloads them or summarizes them first.  Common strategies include *offloading large outputs to storage* (saving tokens by replacing them with references or snippets), *summarizing past messages with an LLM*, and *using retrieval memory* (e.g. vector stores) for long-term facts.  Policies vary: OpenCode, for example, first **marks/hides** old messages beyond a threshold and then uses a hidden “compaction” agent to produce a structured summary, retaining only a concise recap in-context【49†L330-L339】【62†L1597-L1604】. LangChain’s DeepAgents likewise offload >20k-token tool responses to disk and truncate old tool calls when the context hits ~85% capacity【12†L114-L118】【12†L124-L129】. Only when those measures are insufficient does DeepAgents invoke LLM summarization of the conversation【12†L133-L141】. In multi-agent setups (e.g. OpenCode’s agent teams or LangChain’s agent hierarchies), each agent generally maintains its own context; they share information by passing messages or summaries between sessions【56†L55-L64】【49†L355-L364】.

The **token accounting** implications are clear: if a 2,000-token `npm test` output is fed raw into every prompt, it costs ~0.06 USD per 1,000 tokens (using GPT-4 rates) each turn. By contrast, offloading that output to a file (and only injecting a short filename or preview) effectively makes it “free” in future prompts, saving ~2,000 tokens per call. Summarizing that output (say down to 500 tokens) costs an extra summarization API call but saves on-context cost thereafter.  Retrieval (storing embeddings) adds initial embedding costs but allows fetching relevant chunks on demand rather than always including the full output.

This report details these mechanisms, citing official docs and code. A comparison table (below) contrasts systems (OpenCode, LangChain, etc.) on their handling of tool output and summarization. Finally, we list best practices and configurable heuristics (when to trigger summarization, window sizes, summary format).

# Context Summarization in OpenCode

OpenCode (anomalyco/opencode) implements **automatic compaction** of chat context.  When the session token count grows near the model’s limit, OpenCode first *prunes* and then *summarizes*.  Pruning “hides” older messages by stamping them with a `compacted` timestamp (without deleting them from storage) once pruning would free >20,000 tokens【49†L330-L339】.  Crucially, OpenCode protects the most recent ~40,000 tokens and the last 2 user turns from pruning【49†L330-L339】【42†L1578-L1581】. In practice, this means very old tool outputs or assistant messages become invisible to new prompts. The diagram below illustrates how OpenCode marks past messages hidden (tool outputs #5 and #8 become zero-length, i.e. ignored)【59†embed_image】:

【59†embed_image】 *Figure: OpenCode’s pruning step marks old parts of the session “hidden” (timestamped) once token usage is high【49†L330-L339】【42†L1578-L1581】. In this example, outputs of Tool calls #5 and #8 are hidden (0 tokens), preserving recent context for the model.*

If pruning alone does not reduce the active context enough, OpenCode invokes a hidden **compaction agent**. This sub-agent uses the same LLM to generate a structured summary of the conversation so far. The summary is injected *without interrupting the user’s flow* – in fact OpenCode automatically replays the last user message so the AI’s “memory” stays focused on the latest query【49†L355-L364】.  The summary follows a fixed multi-section template: for example, the code specifies headings like “## Goal”, “## Constraints & Preferences”, “## Progress (Done/InProgress/Blocked)”, “## Key Decisions”, “## Next Steps”, etc.【62†L1597-L1604】【62†L1623-L1626】.  An example (from OpenCode’s code) is shown below【62†L1597-L1604】【62†L1623-L1626】:

```

- [single-sentence task summary]

## Constraints & Preferences
- [user constraints or "(none)"]

## Progress
### Done
- [...]
### In Progress
- [...]
### Blocked
- [...]

## Key Decisions
- [...]

## Next Steps
- [...]
```

After summarization, the *entire* conversation history is effectively replaced by this concise summary in active context, but the original full history remains stored off-session if needed. This dual approach means OpenCode preserves only the essentials in-context (reducing tokens drastically) while still allowing the agent to retrieve or audit older details if necessary.  As Justin3go notes, “OpenCode’s approach is the most developer-friendly … it first boxes up old files and labels them (hide), and only summarizes when absolutely necessary”【49†L330-L339】【49†L423-L433】. OpenCode’s design even preserves language context (e.g. summarizing in Chinese if the session was in Chinese), making the summaries user-friendly.

# Tool Outputs and Token Accounting

By default, **all tool outputs that the agent “sees” are counted as context tokens**. In a chat loop, the pattern is: the model generates an assistant message possibly containing a tool call; the tool executes; the result is appended to the message history as a Tool message, then the next LLM call includes that result. Unless specifically trimmed or offloaded, every word of the result adds to prompt length. For example, in a debugging session, several large tool outputs (file contents, test results) might easily occupy 80% of token budget【49†L99-L104】.

To avoid waste, modern agent frameworks *intercept* large tool outputs. LangChain’s DeepAgents, for instance, immediately **offloads any tool response exceeding ~20,000 tokens** to the filesystem, replacing it in the dialogue with a file path and a short preview【12†L114-L118】. In effect, the raw text goes to disk and does *not* count each turn’s context. Similarly, LangChain will truncate old file-edit tool calls (write or patch actions) when the session hits ~85% of the window, since those writes are already persisted on disk【12†L124-L129】. Only when these offloads are insufficient does it ask the LLM to summarize the history.

LangChain’s middleware explicitly supports managing tool outputs. Its **SummarizationMiddleware** will remove older messages (including tool results) once the message history alone exceeds a threshold【14†L64-L73】【16†L240-L248】.  However, by default it only counts the *message buffer* (not system prompts or tool schema). LangChain notes that tool schema and fixed prompt content can themselves cause “token bloat” that summarization won’t fix【14†L78-L87】【14†L143-L152】. To target tool outputs specifically, LangChain offers **ContextEditingMiddleware**. For example, the `ClearToolUsesEdit` action will clear out older ToolMessage entries while keeping a configurable number of recent outputs【16†L230-L238】. This directly addresses the problem: “large ToolMessage content bloating context” can be pruned on the fly【16†L230-L238】.

In short, yes – tool outputs *are* counted unless removed. Best practices include offloading or trimming them, or summarizing them. The cheapest tokens are those never sent: using tools that return *summaries or IDs* (e.g. a file diff instead of full file) can save tokens【16†L257-L264】.

# Context Compression Mechanisms

Across frameworks, several complementary techniques are used to prevent “context rot”:

- **Offloading/Chunking:** As noted, very large outputs (file reads, search results, logs) are moved out of context into storage (filesystem or database). LangChain DeepAgents and OpenCode both do this. After offloading, the prompt includes only a reference (filename or ID) and possibly a short snippet (like the first few lines)【12†L114-L118】. This reduces immediate token use dramatically. For example, an `npm test` output of 2,000 tokens can become a `read_file("log.txt")` tool call instead, with maybe ~20 tokens in context (the path string). Chunking similarly breaks up a big document into vector-store entries or text chunks to retrieve on demand.

- **Memory Retrieval (RAG):** Systems often supplement on-the-fly context with an external memory store. Long-term facts (project specs, design decisions) are vector-embedded so that each turn can retrieve only relevant snippets. Agentic patterns use a combination of *episodic memory* (summarized events) and *long-term memory* (knowledge base). For coding, one might store key functions or error explanations in a vector DB. When the context needs them, a retriever fetches only matching chunks, keeping context small. This trades fewer prompt tokens for the overhead of embedding and search. (For example, storing a 100kb code file as embeddings allows queries on the fly rather than passing the whole file each time.)

- **Summarization/Compression:** When neither offloading nor retrieval frees enough space, the agent compresses by summarizing. OpenCode’s compaction agent, LangChain’s in-context summarization, and Claude Code’s memory all fall here. Summarization might use a structured prompt (see above). In theory, one could also use chunk-wise summarization: break the chat log into blocks, summarize each, then concatenate summaries. Some projects even use rolling summaries (a “memory” blob gradually updated) or hierarchical summarization. The policies for what gets summarized vs retained vary. Codex CLI chooses to keep original user messages verbatim and only summarize assistant content【49†L401-L408】, whereas OpenCode even compresses user requests into the summary (but then re-injects the last user message after summarizing)【49†L399-L408】【49†L430-L434】.

- **Selective Retention:** Most systems preserve recent context intact (the “tail”). OpenCode, for instance, always preserves at least the last 40k tokens and 2 user turns【42†L1578-L1581】【49†L330-L339】. LangChain DeepAgents uses a configurable *preserve_recent_tokens* (default = 25% of usable window) to ensure the freshest interactions stay in memory【42†L1751-L1760】. This tail often includes the last user question and preceding assistant replies, ensuring continuity. Beyond the tail, older assistant replies or tool outputs can be pruned/summarized more aggressively.

- **Incremental Summaries:** Some advanced agents maintain a running summary. For example, OpenCode’s plugin for persistent memory uses a *goal-progress-todo* format and triggers at ~80% usage【21†L130-L138】. After summarizing, the summary is saved as a “memory” and the context is cleared to just the summary plus recent tail. Future turns can use that memory (via retrieval) instead of re-processing old chat. LangChain’s DeepAgents similarly writes the full conversation to storage before summarizing in-context, so that the summary suffices for continuing the session【12†L135-L143】.

# How Multi-Agent Systems Handle Context

In multi-agent workflows (where “primary” agents spawn “subagents” or “teammates”), each agent typically has its own conversation buffer. OpenCode’s agent-teams, for example, runs all teammates in the same process but gives each a separate **session** with its own context window【56†L55-L64】.  They communicate by writing to each other’s “inbox” files or by injecting synthetic user messages【56†L78-L87】.  Importantly, context compression is usually done **per agent**. That is, each agent prunes or summarizes *its own* chat history when needed. If a subagent finishes a task, it might return a summary or result back to the parent agent, but its internal context can then be discarded or summarized. For instance, after a subagent completes code review, its highlights could be returned as a brief report; there is no need to keep its entire log in every other agent’s memory.

In architectures like “agent teams” or LangChain’s AgentChains, agents may share a **knowledge base or vector memory** for persistence. For example, OpenCode agents could write key findings to a shared memory store that all team members can retrieve. However, the core context compression (pruning old conversation turns) remains local.  Since subagents are often short-lived, some systems omit summarization for them entirely, relying on the parent to interpret their outputs. The trade-off in multi-agent setups is that separate contexts use more total tokens (multiple windows), but allow parallelism.  There is no single global context – instead, agents pass messages or summaries to inform each other, then each agent trims its own past.

# Trade-offs: Latency, Accuracy, Cost

Every compression strategy has costs and benefits. **Summarization** preserves information at the cost of extra LLM calls and possible loss of detail. For example, summarizing a 2,000-token test log into 500 tokens might cost ~0.45 USD (depending on model pricing) and add ~500ms latency for the summary call, but saves ~1,500 context tokens (≈$0.045) on subsequent steps. It also risks omitting a specific error message that turns out to be needed later. **Offloading/pointers** (e.g. saving the log to a file) has minimal token cost (only a filename or short snippet is sent) and very low latency overhead for writing to disk. However, it means the LLM no longer has the content unless a tool explicitly retrieves it. If the agent later needs detail, it must use a `read_file` tool or retrieval, incurring an extra round-trip.  **Retrieval via memory** spreads cost out: embedding 2,000 tokens might cost ~0.02 USD (for a small LLM embedder) and storage, then retrieval adds search latency. But queries can fetch just the relevant 100 tokens cheaply. If done well, this yields both low token usage and high fidelity, at the price of system complexity.

In practice, one balances these factors. A simple heuristic: *when a tool output is much larger than needed*, offload and use a brief summary or pointer; *when moderate*, use LLM summarization; *when expected to reuse a fragment many times*, consider vectorizing it. Latency is usually dominated by extra LLM calls or I/O; token cost directly affects API fees. For instance, at GPT-4 pricing (~$0.03/1k tokens), trimming 1,500 tokens saves ~$0.045 per request, which over many requests can offset the cost of one summarization call.

# Example: Handling a 2,000-token Tool Output

Consider an `npm test` producing **2,000 tokens** of log output. Below are three strategies:

- **No compression:** Pass all 2,000 tokens as a ToolMessage each turn. *Cost:* ~2,000 tokens extra per model call (roughly \$0.06 per call at \$0.03/1k). *Latency:* minimal overhead (just insertion). *Downside:* Model context fills quickly; repeating this every turn is costly and wastes tokens after the test is done.

- **Offload to file:** Save the output to `test.log` on disk. In context, include only something like `"Read file test.log"` or a short snippet (say 20 tokens). *Cost:* ~20 tokens (\$0.0006) for the reference. *Latency:* tiny disk I/O. *Trade-off:* The LLM no longer “sees” the log unless explicitly asked. If a later turn needs details, the agent must use a tool to re-read or search the file (adding I/O latency and maybe new tokens).

- **Summarize:** Call the LLM to compress the log. For instance, summarize 2,000 tokens down to ~500 tokens (e.g. highlights of failures). *Cost:* One additional LLM call (~\$0.06 for input + \$0.015 for output). Future context now has 500 instead of 2,000. *Latency:* extra model call (~500ms–1s). *Trade-off:* Saves ~1,500 tokens per turn henceforth (\$0.045 saving per call), but may omit a specific stack trace detail.

- **Retrieval:** Embed the 2,000 tokens in a vector store ahead of time. At each turn, if relevant, retrieve only needed lines (e.g. top 50 tokens). *Cost:* initial embedding cost (~\$0.02) and ~50 tokens per retrieval. *Latency:* vector search (tens of ms) plus additional LLM prompt tokens. *Trade-off:* Good for sparse usage – no need to prompt every turn, but requires building/maintaining a memory DB.

In practice, a hybrid is best. For example, DeepAgents would offload the log file as soon as it’s large, then when context nears limit, summarize the conversation (including a note about the test log). OpenCode would hide that turn entirely if old, or else produce one summary that covers the test results. LangChain middleware might use `ClearToolUsesEdit` to drop the raw log after summarizing it. The cost/latency calculus depends on usage patterns: summarization pays off if multiple future turns refer to the log, whereas offloading is cheapest if detail is seldom needed.

# Recommended Practices and Heuristics

Based on the above, some best-practice guidelines emerge:

- **Trigger summarization early:** Rather than waiting until the context is full, trigger at ~70–80% of model capacity【21†L130-L138】. This “preemptive compaction” avoids losing key information. OpenCode’s plugin suggests 80% threshold by default【21†L130-L138】.

- **Protect the tail:** Always keep the last few interactions verbatim. Typical defaults are to preserve at least the final user question and the assistant’s last answer (or a fixed token budget, e.g. 2,000–8,000 tokens)【42†L1588-L1596】. This ensures continuity. Only older assistant messages should be summarized/hid.

- **Summarize strategically:** Use structured summaries with clear headings (as in OpenCode) to maximize recall of goals and progress. Do not try to have the LLM invent sections on its own; use prompts/templates. The OpenCode example uses bullet sections (“Done/In Progress/Blocked”) which help the model focus. In LangChain, you can provide a templated summarization prompt for consistency. When summarizing, include all original user requests verbatim to prevent “goal drift”【21†L134-L142】.

- **Configure middleware tokens vs messages:** In LangChain’s SummarizationMiddleware, you may want to count actual tokens rather than characters to trigger compression reliably【14†L82-L90】. Also adjust the `keep` parameter (how many messages or tokens to retain) to suit your usage. If tool definitions or system prompts dominate cost, consider other fixes (like LangChain’s LLM tool selector to reduce schema overhead)【14†L146-L153】【16†L247-L257】.

- **Handle system and tool schema:** Remember system prompts and tool function definitions often *aren’t* in the message history counted by middleware【14†L88-L96】. They still consume tokens in the API call. To manage them, limit the number of active tools per turn (use tool selectors) and keep system prompts concise.

- **Combine offload and summarization:** Use offloading for very large chunks (e.g. dumping an entire file) and summarization for narrative history. For instance, after summarizing, immediately clear the full context to the summary only. LangChain’s DeepAgents writes conversation to disk before summarizing, ensuring nothing is lost【12†L141-L149】. OpenCode similarly preserves the old conversation database while using only the summary moving forward.

- **Monitor token usage:** Log the context token count before each call. If using a custom model or provider (like Groq), custom token counters may be needed【14†L122-L131】. Align the agent’s internal counter with the actual API usage to avoid surprises.

- **Memory vs context:** For long-running projects, invest in a memory store. Use embeddings and retrieval to handle static information (project spec, common error patterns, style guides). Keep the LLM context for dynamic conversational state. This hybrid reduces redundant context data.
